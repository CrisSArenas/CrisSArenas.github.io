Problem & Motivation

Portfolio managers at institutional allocators — OCIOs, endowments, family offices — are responsible for monitoring dozens of managers across public and private markets, tracking allocation drift against policy targets, synthesizing quarterly letters and DDQs, keeping up with macro and cross-asset dynamics, and producing written investment memos for committee review. Each of these tasks requires pulling data from different systems, reading unstructured documents, running risk calculations, and composing narratives — a process that is highly manual, time-intensive, and difficult to scale.

A single chatbot or RAG system is insufficient for this workflow. Investment research is inherently multi-step: you need to pull the portfolio data before you can compute drift; you need to read the manager letter before you can assess whether the strategy has changed; you need both the portfolio state and the market context before you can write a meaningful memo. These dependencies require coordination, not just retrieval.

The objective was to build an agentic system — multiple specialized AI agents, each responsible for a distinct domain, orchestrated to execute multi-step investment workflows end-to-end — while maintaining the human-in-the-loop controls and audit trails that institutional investing demands.

Core design constraint: This is a decision-support system, not an autonomous investing system. Every output passes through human review before any action is taken. The system accelerates the PM's workflow — it does not replace the PM's judgment.

System Overview

The Agentic Investment Research & Portfolio Risk Copilot is a multi-agent AI system designed to act as a junior investment team for portfolio managers. Rather than a single monolithic model, the system coordinates five specialized agents that each handle a distinct domain of the investment workflow:

  • Pull holdings, exposures, performance, and risk data from portfolio systems
  • Retrieve manager letters, factsheets, DDQs, and investment notes
  • Monitor macro and market signals for changes relevant to portfolio positioning
  • Identify drift, concentration, liquidity, or regime-related risks
  • Compare current portfolio exposures against targets or policy constraints
  • Produce a draft investment memo or portfolio review for human approval

Each agent has its own tool set, retrieval strategy, and output schema. The orchestration layer decomposes high-level queries (“prepare the weekly portfolio review”) into sub-tasks, routes them to the appropriate agents, resolves inter-agent dependencies, and assembles the final output — with explicit human approval checkpoints at every decision boundary.

Agent Architecture — Deep Dive

The system is structured as five specialized agents, each with distinct tool access, retrieval strategies, and output schemas. The orchestration layer routes queries, manages inter-agent dependencies, and assembles final outputs. Each agent is a self-contained reasoning unit with access to a defined set of tools — it plans its own execution steps, calls tools as needed, and returns a structured output conforming to a typed schema.

1. Data Ingestion Agent

Pulls holdings, exposures, performance, benchmark, and market data from internal files or third-party providers. This agent acts as the data foundation for every downstream workflow — no other agent can operate without a current portfolio snapshot. Handles schema normalization, identifier resolution (ticker → CUSIP/ISIN), currency conversion, and data quality validation.

  • Data Sources: Custodian files (CSV/XLSX), fund admin feeds, Bloomberg/FactSet APIs, internal SQL databases, NAV feeds, capital call/distribution notices
  • Normalization: Schema alignment via Pydantic models, asset identifier resolution (ticker → CUSIP/ISIN → internal ID), currency conversion to base currency, mapping to canonical taxonomy (asset class, sub-strategy, vehicle type)
  • Validation: NAV reconciliation (aggregated MV vs reported NAV, threshold ≤ 10bps), missing data detection, stale price flagging, duplicate holding detection
  • Output: Normalized portfolio snapshot with holdings, weights, exposures, NAVs, performance attribution, and data freshness metadata
  • Tools: SQL query executor, CSV/XLSX parser, REST API connectors, Pydantic schema validators, FX rate service
Data Ingestion Agent — Tool Execution Flow:

  query("get current portfolio for Client X")
      │
      ├── tool: sql_query → fetch holdings from portfolio DB
      ├── tool: api_call  → pull latest NAVs from fund admin
      ├── tool: csv_parse → ingest custodian reconciliation file
      ├── tool: fx_rates  → convert non-USD positions to base currency
      ├── tool: id_resolve → map tickers to CUSIP/ISIN
      └── tool: validate  → run reconciliation checks
      │
      └── output: PortfolioSnapshot (typed JSON schema)

2. Manager Research Agent

Reads manager letters, DDQs, factsheets, and prior investment memos, then extracts relevant changes in positioning, performance, liquidity terms, or strategy. This is the most retrieval-intensive agent — it operates over a vectorized document store containing years of manager correspondence and performs multi-step retrieval to build a comprehensive view of each manager’s current state.

  • Document Types: Quarterly letters, DDQs, factsheets, IC memos, side letters, subscription docs, annual audited financials
  • Extraction Targets: Strategy changes (style drift, new market entry), AUM movements (inflows/outflows, capacity constraints), team changes (key person departures), liquidity term modifications (gate changes, lock-up extensions), performance vs peers (quartile ranking, benchmark-relative returns), fee structure changes
  • Multi-Step Reasoning: The agent first retrieves the most recent letter for a given manager, identifies what changed since the prior period, then cross-references against the DDQ and IC memo to flag inconsistencies or material shifts
  • Tools: PDF/DOCX parser (layout-aware), vector store retriever (embeddings + hybrid search), structured extraction chains (LLM-powered field extraction), document comparator (diff between periods)
Manager Research Agent — Multi-Step Retrieval:

  query("summarize changes for Manager ABC since last quarter")
      │
      ├── step 1: retrieve latest quarterly letter (Q4 2025)
      ├── step 2: retrieve prior quarterly letter (Q3 2025)
      ├── step 3: extract structured fields from both
      │           (AUM, strategy, positioning, liquidity, performance)
      ├── step 4: diff extraction — identify material changes
      ├── step 5: cross-reference against DDQ (verify terms unchanged)
      ├── step 6: pull peer performance from data ingestion agent
      └── step 7: assemble manager research summary with citations
      │
      └── output: ManagerResearchSummary (typed JSON + source refs)

3. Risk Monitoring Agent

Checks for allocation drift, concentration, style shifts, liquidity mismatches, factor exposure changes, and unusual drawdown or volatility behavior. This agent operates on the portfolio snapshot from the Data Ingestion Agent and compares the current state against IPS targets, policy constraints, and historical norms. It produces a structured risk report with severity-classified alerts.

  • Drift Analysis: Current weights vs. SAA/TAA targets at every level of the allocation hierarchy (asset class, sub-strategy, manager, vehicle). Tolerance bands are configurable per level — tighter at the asset class level, wider at the manager level
  • Concentration Checks: Single manager concentration, sector/geography concentration, factor concentration (beta, duration, credit quality), vintage year concentration for private markets
  • Liquidity Analysis: Redemption terms vs. allocation weight, gate provisions and lock-up schedules, side pocket exposure, capital call commitments vs. available liquidity, aggregate liquidity tier breakdown (daily, monthly, quarterly, illiquid)
  • Style Drift Detection: Rolling factor exposure analysis (value, momentum, quality, size), comparison to stated mandate and historical factor profile, flagging of statistically significant exposure changes
  • Drawdown & Volatility: Manager-level drawdown monitoring vs. historical max drawdown, rolling volatility vs. expected range, return distribution anomaly detection (unusual skew or kurtosis changes)
  • Capital Pacing: Capital call/distribution modeling across PE/PC vintages, cash flow adequacy projections, J-curve position tracking per vintage

4. Market Context Agent

Monitors macro and cross-asset signals such as spreads, rates, volatility, momentum, and credit conditions, then explains what changed and why it matters for the portfolio. This agent provides the macro narrative layer that contextualizes portfolio-level observations — turning raw data points into investment-relevant insight.

  • Credit Signals: IG and HY spreads (OAS), spread rate-of-change, cross-quality spread ratios, distressed debt indicators, new issuance activity
  • Rates & Curve: Yield curve shape (2s10s, 2s30s), real yield levels, forward rate expectations, rate volatility (MOVE index), central bank policy signals
  • Volatility: VIX levels and term structure, realized vs. implied vol spread, cross-asset vol correlations, skew dynamics, VVIX (volatility of volatility)
  • Momentum: Cross-asset trend signals across multiple horizons (1M, 3M, 6M, 12M), breadth measures, sector rotation indicators, relative strength rankings
  • Macro: PMI/ISM and direction of change, employment data, inflation expectations (breakevens), economic surprise indices, leading indicator composites
  • Flows: Fund flow data across asset classes, ETF creation/redemption activity, safe-haven demand, positioning indicators (CFTC, put/call ratios)

The agent integrates with the ML Regime Classification System (separate production model) to incorporate the current regime label and confidence score into its macro narrative, providing continuity across systems.

5. Memo Generation Agent

Combines outputs from all other agents into a clean draft for a weekly PM note, IC memo, or manager review summary. This is the final-mile agent — it consumes structured outputs from every upstream agent and synthesizes them into a coherent narrative that a portfolio manager can review, edit, and approve for distribution. Every claim is cited back to its source agent and underlying data.

  • Output Formats: Weekly PM note, IC memo, manager review summary, risk alert brief, ad-hoc research response
  • Template System: Jinja-based templates per memo type, with dynamic sections that expand or contract based on the volume of material from upstream agents
  • Citation Architecture: Every numerical claim, risk flag, and manager observation is tagged with a source reference (agent name, data timestamp, document source). Citations render as inline footnotes in the final output
  • Multi-Stage LLM Chain: Draft generation → tone/compliance review → numerical accuracy check → formatting pass
  • Guardrails: No fabricated data, all numbers traced to source, compliance-safe language, institutional tone, explicit uncertainty language where confidence is low
Memo Generation — Multi-Stage LLM Chain:

  Stage 1: Draft Generation

  Stage 2: Tone & Compliance Review
      Input: draft from Stage 1

      Input: revised draft + original structured data

  Stage 4: Formatting & Citation
      Input: validated draft
      Task: apply template formatting, render citations as footnotes
      Method: Jinja template engine + citation linker
      Output: final memo (HTML/PDF) ready for PM review

Inter-Agent Communication Protocol

Agents communicate exclusively through structured JSON payloads conforming to typed schemas. This ensures composability — any agent can consume any other agent’s output without brittle string parsing.

  • Each agent output includes metadata: source agent, execution timestamp, data freshness, confidence level, data lineage references
  • The orchestrator maintains a shared context store (key-value) accessible to all agents within a workflow execution
  • Agents can request data from other agents mid-execution via the orchestrator (e.g., Risk Agent requests additional market context if a novel signal pattern is detected)
  • All inter-agent messages are logged for audit and debugging

Error Handling & Resilience

  • Retry with backoff: Tool call failures (API timeouts, DB connection drops) trigger automatic retries with exponential backoff before surfacing to the orchestrator
  • Graceful degradation: If the Market Context Agent fails (e.g., market data feed outage), the workflow proceeds with a “market context unavailable” flag rather than blocking the entire pipeline
  • Partial output: The Memo Generation Agent can produce partial memos when upstream data is incomplete, with explicit “data pending” placeholders and a human notification
  • Timeout guards: Each agent has a maximum execution time; if exceeded, current progress is saved and the agent reports partial results
  • Idempotency: Workflow re-runs produce the same result given the same input state, enabling safe retries

Human-in-the-Loop Controls

Every workflow includes explicit approval checkpoints. The system never takes autonomous action on portfolios — no trades, no communications, no escalations without human sign-off.

  • Draft memos require PM review and sign-off before distribution
  • Risk alerts are surfaced for human triage, not auto-escalated to clients or committees
  • Manager research extractions are flagged for analyst verification, especially on material changes (AUM, strategy, team)
  • All agent outputs include an audit trail with full data provenance
  • The PM can override, edit, or reject any agent output at any stage
  • Approval history is logged and timestamped for compliance records

RAG & Document Retrieval Architecture

Document Processing Pipeline

Manager documents (letters, DDQs, factsheets) are ingested through a multi-stage pipeline designed to preserve document structure — including tables, headers, footnotes, and section boundaries — which are critical for accurate information extraction in financial documents.

  1. Ingestion — PDF/DOCX parsing with layout-aware extraction using document AI models that preserve table structure, column alignment, and reading order
  2. Pre-processing — Header/footer removal, page number stripping, boilerplate detection (disclaimers, legal notices), table extraction into structured format
  3. Chunking — Semantic chunking with configurable overlap, preserving section boundaries and table integrity. Tables are kept as atomic chunks to prevent splitting structured data across retrieval boundaries
  4. Embedding — Dense vector embeddings generated per chunk, stored in a vector database with associated metadata
  5. Indexing — Rich metadata tagging: manager name, fund name, document type, publication date, vintage year, section type (performance, strategy, liquidity, outlook)

Hybrid Retrieval Strategy

Pure vector similarity search alone is insufficient for investment documents — a query about “Manager ABC’s liquidity terms” should never return chunks from Manager DEF, regardless of semantic similarity. The retrieval pipeline combines:

  • Metadata pre-filtering: Hard filters on manager name, document type, and date range before vector search executes — ensures results come from the correct document universe
  • Vector similarity search: Dense embedding similarity within the filtered set to find semantically relevant passages
  • Keyword matching: BM25-style sparse retrieval as a complementary signal, especially useful for specific terms (fund names, ISIN codes, fee percentages) that embeddings may not capture precisely
  • Cross-encoder re-ranking: Top-K results from combined retrieval are re-ranked by a cross-encoder model that scores query-passage relevance with higher accuracy than bi-encoder similarity alone
  • Reciprocal Rank Fusion: Combining ranked results from vector and keyword retrievers into a unified ranking before re-ranking

Context Window Management

Investment documents can be long (50+ page DDQs, 20+ page quarterly letters). Effective retrieval requires maximizing the relevance density of the context window passed to the LLM:

  • Relevance-ranked chunk ordering — most relevant chunks first in the context window
  • Deduplication — overlapping chunks from the same document section are merged
  • Dynamic context budget — allocates more context window space to the primary document being analyzed, less to supporting references
  • Source attribution preserved on every chunk — document name, page number, section header — enabling citation in downstream memo generation
RAG Pipeline — End-to-End Flow:

  New Document Arrives (e.g., Manager ABC Q4 2025 Letter)
      │
      ├── PDF parser → layout-aware text + table extraction
      ├── Pre-processor → remove boilerplate, extract tables
      ├── Chunker → semantic chunks with section metadata

Prompt Engineering & LLM Chain Design

Agent-Level Prompt Architecture

Each agent uses a carefully structured system prompt that defines its role, available tools, output schema, and behavioral constraints. Prompts are Jinja-templated to inject dynamic context (portfolio name, date range, specific manager) at execution time.

System Prompt Structure (Per Agent)

  • Role definition: “You are the Risk Monitoring Agent for an institutional OCIO platform...”
  • Tool descriptions: Typed descriptions of each tool the agent can call, including input/output schemas
  • Output schema: Explicit JSON schema the agent must conform to, enforced via structured output parsing
  • Behavioral constraints: “Never fabricate data. Never invent holdings or returns. If data is unavailable, say so explicitly.”
  • Domain vocabulary: Investment-specific terminology definitions to ensure consistent usage (e.g., “drift” = weight deviation from SAA target, not style drift)
  • Examples: Few-shot examples of correct tool usage and output formatting for the agent’s specific domain

Memo Generation — Multi-Stage Chain

The Memo Generation Agent uses a four-stage LLM chain where each stage has a distinct system prompt, temperature setting, and validation step:

  • Stage 1 — Draft (temp=0.2): Synthesize structured agent outputs into a narrative. Prioritize clarity and accuracy over style. Include all relevant data points
  • Stage 2 — Tone (temp=0.1): Adjust language to institutional standards. Remove informal phrasing, ensure balanced and measured commentary, add appropriate caveats and uncertainty language
  • Stage 3 — Accuracy (temp=0.0): Cross-check every number against source data. Flag any discrepancy. This stage uses deterministic settings to ensure reproducibility
  • Stage 4 — Format (deterministic): Apply Jinja template formatting, render citations as footnotes, generate table of contents for longer memos

Guardrail Architecture

LLM outputs in institutional finance carry real risk — a fabricated return number or misattributed risk flag could influence investment decisions. The guardrail system operates at multiple levels:

  • Input guardrails: Sensitive data (client names, account numbers, SSNs) are redacted or tokenized before reaching the LLM. Tokenized identifiers are re-mapped in post-processing
  • Behavioral guardrails: System prompts include explicit “never” rules — never fabricate data, never provide investment advice, never speculate on future returns without labeling as forward-looking
  • Output validation: Regex extraction of all numerical values in generated text, cross-referenced against source data. Any mismatch triggers a flag for human review rather than silent pass-through
  • Semantic consistency: Generated narrative sentiment is compared against the structured data direction (e.g., the text should not describe performance as “strong” if the return is negative)
  • Compliance filters: Output scanned for regulatory-sensitive language (forward-looking statements, guarantees, superlatives about performance) and flagged or rewritten

Risk Detection Framework

Monitoring Dimensions

The Risk Monitoring Agent evaluates the portfolio across six orthogonal risk dimensions, each with its own threshold logic, severity classification, and escalation path.

1. Allocation Drift

  • Current weights vs. SAA/TAA targets at every hierarchy level (asset class → sub-strategy → manager → vehicle)
  • Configurable tolerance bands per level: ±2% at asset class, ±3% at sub-strategy, ±5% at manager
  • Trend analysis: is drift increasing, stable, or correcting? Consecutive-month drift acceleration triggers higher severity
  • Rebalancing cost estimation: projected transaction costs and tax implications of correcting the drift

2. Concentration Risk

  • Single manager: flag if any single manager exceeds N% of total portfolio (configurable per asset class)
  • Sector/geography: look-through concentration analysis aggregated across all managers
  • Factor: aggregate factor exposure (beta, duration, credit quality) checked against policy limits
  • Vintage year: private market vintage concentration — flag if capital deployed is skewed to ≤2 vintages

3. Liquidity Mismatch

  • Aggregate liquidity tiering: % of portfolio available at T+1, T+30, T+90, T+365, illiquid
  • Comparison against policy minimum liquidity requirements
  • Gate and lock-up tracking: flag managers approaching gate thresholds or lock-up expiry
  • Stress scenario: if 10% of liquid assets needed for capital calls, does remaining liquidity still meet policy?

4. Style Drift

  • Rolling factor exposure analysis (60-day, 120-day windows) vs. mandate definition
  • Statistical significance testing: is the exposure change within normal variation or a genuine shift?
  • Cross-reference with Manager Research Agent: does the manager letter acknowledge the positioning change?

5. Regime Sensitivity

  • Integration with ML Regime Classification System: when regime shifts, re-evaluate portfolio positioning
  • Conditional risk metrics: expected portfolio behavior under current regime vs. alternative regimes
  • Regime-transition impact: estimated portfolio drag if current positioning is misaligned with emerging regime

6. Capital Pacing

  • Capital call/distribution schedule modeling across all PE/PC/real asset commitments
  • Cash flow adequacy projection: expected calls vs. available liquidity over 4-quarter forward window
  • J-curve position tracking: where each vintage sits on its lifecycle curve
  • Over-commitment analysis: total unfunded commitments vs. NAV, with stress scenarios

Alert Classification & Escalation

Every alert is classified by severity and routed accordingly:

  • Informational: No action required, logged for awareness. Example: allocation drift within tolerance but trending toward breach
  • Warning: Breach detected, PM review recommended. Example: single manager concentration exceeds soft limit
  • Action Required: Material breach or multi-dimension risk convergence. Example: liquidity shortfall projected under capital call stress scenario

Alerts at all levels are surfaced to the PM for human triage — the system never auto-escalates to clients, committees, or external parties.

Security & Compliance

Data Security

  • All portfolio and client data encrypted at rest (AES-256) and in transit (TLS 1.3)
  • Role-based access control with principle of least privilege — agents have read-only access to source systems
  • No write-back to portfolio systems — the agentic layer is read-only; all actions require human execution
  • Sensitive data redaction before LLM inference — no PII, client identifiers, or account numbers sent to model endpoints
  • Tokenized security identifiers in LLM context; re-mapped in post-processing
  • Secrets managed via AWS Secrets Manager / Azure Key Vault with automatic rotation
  • Network-level protection: VPC subnets, security groups, private endpoints for all data connections

LLM Guardrails

  • Deterministic temperature settings (0.0–0.2) for reproducible outputs
  • Explicit constraints embedded in system prompts: never fabricate data, never invent holdings or returns, never provide investment recommendations without human approval
  • Post-generation validation: regex + semantic checks for numerical accuracy against source data
  • Compliance-safe language filters aligned with institutional communication standards and regulatory requirements
  • Forward-looking statement detection and labeling
  • Output hash logging for reproducibility verification

Audit & Governance

  • Full audit trail: every agent action, tool call, retrieval, and generation logged with timestamps and execution context
  • Data lineage tracking from source system to final memo output — every number is traceable to its origin
  • Human approval records with timestamps, reviewer identity, and any edits made
  • Model version pinning: every output records the exact model version, prompt version, and agent version used
  • Aligned with SOC 2 Type II internal controls and NIST 800-53 framework
  • Retention policies aligned with regulatory requirements (7-year minimum for investment communications)

Operational Controls

  • Agent processes have read-only access to all upstream data sources — no ability to modify portfolio systems, execute trades, or send communications
  • Rate limiting on LLM API calls to prevent runaway costs
  • Circuit breaker pattern: if an agent produces anomalous outputs (e.g., portfolio weights summing to >110%), the workflow halts and alerts
  • SLA monitoring: weekly review workflow must complete within defined time window

Performance & Scalability

Execution Performance

  • Full weekly portfolio review workflow: completes in under 5 minutes per portfolio (including all agent execution, retrieval, and memo generation)
  • Phase 1 (data collection) agents run in parallel, reducing wall-clock time by ~60% vs. sequential execution
  • RAG retrieval latency: <2 seconds per query (including metadata filtering, vector search, and re-ranking)
  • Memo generation (4-stage chain): ~30 seconds per memo

Scalability Architecture

  • Stateless agent execution — each workflow run is independent, enabling horizontal scaling across portfolios
  • Multiple portfolio reviews can run concurrently with no shared state conflicts
  • Vector database scales independently of agent compute — document corpus growth does not degrade query latency
  • Airflow workers scale via Kubernetes executor for data pipeline tasks

Throughput

  • Designed to support 20+ portfolio reviews per day in batch mode
  • Ad-hoc queries (single manager research, specific risk check) execute in under 60 seconds
  • Document store handles 10,000+ documents across 100+ managers with sub-second retrieval

Impact

The system enables portfolio managers to:

  • Accelerate research workflows — multi-step retrieval and synthesis across holdings, market data, and manager documents in minutes instead of hours. Weekly portfolio review preparation reduced from 4+ hours of manual work to a 5-minute automated draft with PM review
  • Surface risks proactively — automated monitoring of drift, concentration, liquidity, and regime changes with configurable alert thresholds. Risk flags that previously required manual calculation are now surfaced automatically with supporting data and trend context
  • Standardize memo quality — consistent, citation-backed drafts for PM notes, IC memos, and manager reviews. Every claim is traceable to its source, eliminating the risk of undocumented assertions in investment communications
  • Maintain institutional controls — human-in-the-loop approval at every decision point with full audit trail. The system augments PM judgment without replacing it, operating within fiduciary constraints
  • Scale coverage — monitor more managers and mandates without proportional headcount increase. The same analyst team can oversee a larger portfolio universe with consistent quality
  • Improve decision speed — when a regime shift or manager event occurs, the system can produce a focused brief with full context in minutes, enabling faster response times on time-sensitive allocation decisions
