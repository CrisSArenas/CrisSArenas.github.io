Real-Time Sensor Fusion & Balance Controller

2025
Real-Time Control

A self-balancing platform built on ESP32 running FreeRTOS. Fuses MPU-6050 gyroscope and accelerometer data with HC-SR04 ultrasonic distance readings through complementary and Kalman filters. Dual PID control loops running at 100 Hz (angle) and 20 Hz (position). Under 2 degrees steady-state error, validated with logic analyzer over extended soak testing.

C++ FreeRTOS ESP32 MPU-6050 HC-SR04 PID Control Kalman Filter

System Overview

This project implements a self-balancing platform that maintains vertical stability using real-time sensor fusion and cascaded PID control. The system runs on an ESP32 microcontroller with FreeRTOS for deterministic task scheduling.

The core challenge is maintaining balance despite noisy sensor data, mechanical vibrations, and external disturbances. This is solved through a multi-stage pipeline:

  • High-rate IMU sampling (gyroscope + accelerometer) via I2C
  • Sensor fusion using both complementary and Kalman filters
  • Cascaded PID control: inner loop (angle) at 100 Hz, outer loop (position) at 20 Hz
  • Ultrasonic distance sensing for position feedback
  • Motor actuation via PWM through H-bridge driver

Hardware Integration

MPU-6050 вҖ” Inertial Measurement Unit

The MPU-6050 provides 3-axis gyroscope and 3-axis accelerometer data over I2C at 400 kHz. The raw data is read at 1 kHz and buffered for the fusion task:

  • Gyroscope: angular velocity (degrees/second) вҖ” fast response, but drifts over time
  • Accelerometer: gravity vector вҖ” no drift, but noisy and sensitive to vibration
  • I2C reads: burst read of 14 bytes (accel XYZ + temp + gyro XYZ) to minimize bus transactions
  • Calibration: offset and scale correction computed at startup from 1000-sample average at rest

HC-SR04 вҖ” Ultrasonic Distance Sensor

The HC-SR04 provides distance measurements via time-of-flight pulse timing. A GPIO trigger pulse initiates a measurement, and the echo pulse width encodes the distance. Readings are filtered with a 5-sample moving median to reject outliers from multipath reflections.

Motor Drive

Dual DC motors controlled through an L298N H-bridge with ESP32 PWM outputs. The LEDC peripheral generates 20 kHz PWM (above audible range) with 10-bit resolution (1024 steps). Direction is controlled by separate GPIO pins for each motor.

Sensor Fusion

Complementary Filter

The complementary filter combines gyroscope integration (high-pass) with accelerometer angle estimation (low-pass) using a tunable alpha parameter:

angle = alpha * (angle + gyro_rate * dt) + (1 - alpha) * accel_angle

With alpha = 0.98, the filter trusts the gyroscope for fast dynamics while using the accelerometer to correct long-term drift. Effective and computationally cheap.

Kalman Filter

A 2-state (angle, angular rate) Kalman filter provides optimal sensor fusion by modeling the system dynamics and measurement noise explicitly:

  • Predict step: Propagates angle state using gyroscope rate and process noise covariance (Q)
  • Update step: Corrects prediction using accelerometer measurement and measurement noise (R)
  • Tuning: Q and R are tuned experimentally вҖ” Q reflects gyro drift rate, R reflects accel noise level
  • Result: Smoother output than the complementary filter with better transient response

Both filters run in parallel during development, with the Kalman filter output used for the production control loop.

Control System

Cascaded PID Architecture

The system uses two nested PID loops:

Inner Loop вҖ” Angle Control (100 Hz)

  • Input: filtered tilt angle from Kalman filter
  • Setpoint: target angle from outer loop (or 0 for pure balancing)
  • Output: motor PWM command (proportional to corrective torque)
  • Tuning: Kp = 45, Ki = 0.5, Kd = 18 (Ziegler-Nichols initial, then manual refinement)
  • Anti-windup: integral term clamped to prevent saturation buildup

Outer Loop вҖ” Position Control (20 Hz)

  • Input: filtered distance from HC-SR04 ultrasonic sensor
  • Setpoint: target position (distance from reference wall)
  • Output: target tilt angle for inner loop
  • Tuning: Kp = 0.8, Ki = 0.02, Kd = 0.3
  • Output clamp: target angle limited to +/- 15 degrees for safety
Result: Under 2 degrees steady-state angle error. Recovery from 10-degree push disturbance in under 800ms with minimal overshoot.

RTOS Architecture

FreeRTOS Task Design

The system is structured as four FreeRTOS tasks with strict priority ordering:

  • SensorRead (Priority 4): 1ms period. Reads MPU-6050 over I2C, writes to shared buffer with mutex protection.
  • SensorFusion (Priority 3): 10ms period. Runs complementary + Kalman filter, publishes filtered angle.
  • AngleControl (Priority 2): 10ms period. Inner PID loop, outputs PWM commands to motors.
  • PositionControl (Priority 1): 50ms period. Outer PID loop, sets target angle for inner loop.

Timing Guarantees

FreeRTOS tick rate is set to 1 kHz for millisecond-resolution scheduling. The sensor read task runs at highest priority to ensure consistent IMU sampling. Task worst-case execution times are validated via logic analyzer:

  • SensorRead: 0.4ms WCET (dominated by I2C transfer time)
  • SensorFusion: 0.1ms WCET (Kalman filter matrix operations)
  • AngleControl: 0.05ms WCET (PID computation)
  • Total CPU utilization: ~12% at steady state

Validation & Testing

Logic Analyzer Verification

Task timing and I2C bus transactions are validated using a Saleae logic analyzer:

  • I2C clock frequency confirmed at 400 kHz
  • Task period jitter measured at less than 50 microseconds
  • No I2C bus contention or NACK errors over 24-hour soak test
  • PWM frequency stable at 20 kHz +/- 0.1%

Extended Soak Testing

The system runs continuous balancing for 24+ hours to validate:

  • No gyro drift accumulation (Kalman filter corrects continuously)
  • No memory leaks (FreeRTOS heap usage stable)
  • No task deadline misses (verified via GPIO toggle instrumentation)
  • Consistent control performance over temperature variation
Key metrics: Under 2 degrees steady-state error, 800ms disturbance recovery, zero missed deadlines over 24-hour test, 12% CPU utilization.
