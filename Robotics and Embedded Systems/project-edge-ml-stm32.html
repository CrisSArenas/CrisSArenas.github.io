Edge ML Inference on STM32

2025 - 2026
Embedded ML

Deployed a quantized image classification model on an STM32H7 microcontroller (512KB RAM) using TensorFlow Lite for Microcontrollers. Wrote bare-metal C drivers for OV7670 camera capture over DCMI with DMA transfers and SPI display output. Direct register access throughout — no HAL abstraction. Full pipeline from model training to INT8 quantization to on-board validation.

TF Lite Micro STM32H7 Bare-Metal C OV7670 DCMI DMA SPI Display INT8 Quantization

System Overview

This project demonstrates end-to-end on-device machine learning inference on a resource-constrained microcontroller. The STM32H7 captures images from an OV7670 camera, runs a quantized CNN classifier using TensorFlow Lite for Microcontrollers, and displays results on an SPI-connected LCD.

The entire system runs on bare metal — no operating system, no HAL library. All peripheral configuration (clocks, GPIO, DCMI, DMA, SPI) is done through direct register manipulation, giving full control over timing, memory layout, and power consumption.

  • Image capture: OV7670 camera via DCMI with DMA double-buffering
  • Inference: TensorFlow Lite Micro with INT8 quantized model
  • Output: Classification results on SPI TFT display
  • Constraint: Everything fits in 512KB SRAM + 2MB Flash

Hardware Layer

STM32H7 Microcontroller

The STM32H743 runs at 480 MHz with a Cortex-M7 core featuring an FPU and DSP instructions. All clock tree configuration (HSE, PLL1, PLL3 for DCMI pixel clock) is done by writing directly to RCC registers. No HAL or LL drivers are used — this provides precise control over peripheral timing and eliminates abstraction overhead.

OV7670 Camera — DCMI Interface

The OV7670 is configured over SCCB (I2C-like) for QVGA resolution (160x120) in RGB565 format. The DCMI peripheral captures frames using its hardware synchronization (VSYNC, HSYNC, PIXCLK). Frame data is transferred directly to SRAM via DMA in double-buffer mode, allowing one buffer to be processed while the other receives the next frame.

  • DCMI configured for continuous capture mode
  • DMA2 Stream 1 with double-buffering (Memory0 / Memory1)
  • Frame complete interrupt triggers inference pipeline
  • SCCB register writes for exposure, gain, and white balance

SPI Display

A 1.8" TFT display connected over SPI1 at 40 MHz. The display driver handles initialization sequences (ST7735 controller), window addressing, and bulk pixel transfers via DMA-assisted SPI TX. Classification results are rendered as text overlays on the live camera feed.

Firmware Design

Bare-Metal Architecture

The firmware is structured as a super-loop with interrupt-driven DMA transfers:

  • Startup: Clock tree init, GPIO alternate function mapping, peripheral enable
  • Camera init: SCCB register configuration, DCMI + DMA setup, capture start
  • Main loop: Wait for frame-complete flag, run preprocessing, invoke TFLite, update display
  • Interrupts: DMA transfer complete (buffer swap), SysTick (timing)

Direct Register Access

Every peripheral is configured through direct register writes. Examples:

  • RCC->AHB1ENR for enabling GPIO and DMA clocks
  • DCMI->CR for capture mode, crop, and embedded sync configuration
  • DMA2_Stream1->CR for channel selection, direction, and double-buffer enable
  • SPI1->CR1 for baud rate, CPOL/CPHA, and DMA TX enable

This approach eliminates HAL overhead (function call chains, error checking for impossible states) and provides deterministic timing for the real-time capture pipeline.

ML Pipeline

Model Training (Host)

The classification model is a lightweight CNN trained in TensorFlow/Keras on a host machine. Architecture is optimized for microcontroller deployment:

  • Depthwise separable convolutions to reduce parameter count
  • Small input resolution (48x48 grayscale)
  • Total parameters under 50K for minimal memory footprint
  • Training with data augmentation (rotation, flip, brightness) for robustness

INT8 Quantization

Post-training quantization converts all weights and activations from float32 to int8:

  • Representative dataset calibration for accurate activation ranges
  • Weights: float32 → int8 (4x size reduction)
  • Activations: quantized at inference time using calibrated scale/zero-point
  • Model size: ~120KB (down from ~480KB float32)
  • Accuracy loss: less than 2% compared to float32 baseline

TF Lite Micro Deployment

The quantized .tflite model is converted to a C byte array and linked into firmware. TF Lite Micro runs inference using a statically allocated tensor arena (configurable size). The interpreter is initialized once at startup and reused for each frame.

Memory budget: ~120KB model + ~80KB tensor arena + ~76KB frame buffers + ~20KB firmware code = ~296KB total, fitting within the 512KB SRAM constraint.

All buffers are statically allocated at compile time with explicit linker section placement. No dynamic memory allocation (malloc/free) is used at runtime, eliminating fragmentation risks and ensuring deterministic behavior.

Performance & Validation

System performance metrics:

  • Inference time: ~45ms per frame (INT8 on Cortex-M7 at 480 MHz)
  • Frame rate: ~15 FPS end-to-end (capture + preprocess + infer + display)
  • Classification accuracy: 94% on test set (less than 2% drop from float32)
  • Power consumption: ~350mW during active inference
  • Memory usage: ~296KB of 512KB SRAM utilized

Validation was performed by comparing on-device inference results against the host TensorFlow model across 500+ test images, confirming bit-exact INT8 agreement and verifying no quantization artifacts in edge cases.
