System Overview

The AI/ML Portfolio Oversight System is a modular, distributed analytical pipeline built to automate institutional portfolio oversight workflows. The system integrates:

  • Structured data ingestion
  • Automated exposure/risk computation
  • Statistical anomaly detection
  • LLM-based narrative generation
  • Pipeline orchestration and dependency management via Airflow

The architecture is designed for fault-tolerant, reproducible, and auditable execution, supporting daily cycles of multi-portfolio exposure monitoring.

Data Engineering Architecture

Data Sources

Data is ingested from multiple structured and semi-structured sources:

  • SQL databases (portfolio exposures, holdings, NAVs)
  • Vendor benchmark APIs (MSCI, Bloomberg style factors)
  • Custodian flat files (CSV, XLSX)
  • Internal performance reports
  • Factor model datasets (HML, Momentum, Quality, Term structure)

All sources land in a staging layer, stored in Parquet for columnar compression, schema evolution, and fast downstream reads.

Data Normalization Pipeline

The pipeline enforces:

  • Schema alignment (via Pydantic models)
  • Mapping to canonical taxonomies (POD, asset class, vehicle structure)
  • Asset identifier resolution (ticker → CUSIP/ISIN mapping)
  • Currency normalization
  • Missing value imputation (deterministic rules, not ML)
  • Reconciliation (NAV vs aggregated MV difference threshold ≤ 10bps)

Failures trigger Airflow task retries and eventually route to error logs.

Exposure & Risk Engine

Factor Exposure Calculation

Uses:

  • PCA-derived market factors
  • Bloomberg-style factor mapping
  • Style factor decomposition (Value, Momentum, Quality, Size)
  • Duration and convexity for fixed income exposures
  • Liquidity tiers (T+1, weekly, monthly, quarterly)

Geographic + Sector Exposures

Computed using:

  • Hierarchical grouping from security metadata
  • Propagated exposures for nested fund-of-funds
  • Weight normalization after look-through

Risk Metrics

Using:

  • Rolling standard deviation (30, 60, 90 days)
  • Correlation matrices
  • Cross-asset beta mapping
  • Idiosyncratic vs systematic risk contribution

Machine Learning & Anomaly Detection

Techniques Used

The system uses a hybrid ensemble of:

  • Isolation Forest (outlier detection)
  • Local Outlier Factor (LOF)
  • Rolling z-score deviation
  • Cluster-based deviation detection (K-Means or hierarchical clustering)
  • Autoencoder reconstruction error (optional for dense datasets)

Input Features

Key engineered features include:

  • Exposure deltas (1d, 5d, 30d)
  • Drift velocity and acceleration
  • Factor loadings
  • Liquidity mismatch indicators
  • Performance drivers vs expected factor returns
Result: Improved anomaly detection, enabling early identification of unusual portfolio exposures and potential risk indicators.

LLM Oversight Engine

Pipeline Structure

Multi-step chain:

  1. Structured Feature Extraction - exposure deltas, risks, anomalies → JSON bundle
  2. Template Selection - Based on portfolio type (OCIO, endowment, DMA, etc.)
  3. LLM Prompt Assembly - Jinja-based template combining numerical insights & qualitative rules
  4. LLM Inference - Via OpenAI or Azure endpoints (temperature = 0 for deterministic results)
  5. Post-Processing & Validation - Regex + semantic checks to prevent hallucinations

Prompt Engineering Design

Prompts include:

  • Structured tables
  • Explicit constraints
  • "Never fabricate data" rules
  • Portfolio-specific vocabulary
  • Risk flags sections

This ensures outputs meet institutional standards.
